# Cache Alignment Filter — 缓存对齐器设计文档

## 背景

DeepSeek v4 Flash API 的缓存命中价（$0.0028/1M tokens）与未命中价（$0.14/1M tokens）相差 **50 倍**。
每次前缀哈希不一致 → 缓存 MISS → 支付全价。

## DeepSeek 缓存判定逻辑（2026年5月官方文档确认）

### 缓存单元

缓存是**离散的前缀单元（cache prefix unit）**——不是逐块哈希，不是模糊匹配，而是**完整的、离散的**前缀单元完全匹配。

### 三种缓存持久化时机

1. **请求边界持久化** — 每次请求在「用户输入结束位置」和「模型输出结束位置」各产生一个缓存前缀单元
2. **公共前缀检测持久化** — 系统自动检测多个请求的公共前缀，将其持久化为独立缓存单元
3. **固定 token 间隔持久化** — 长输入按固定 token 间隔切出缓存单元

### 命中 vs 未命中

- `A + B` 首次 → MISS（写缓存）
- `A + B + C` 第二次 → 完全匹配 `A + B` → **缓存 HIT**，仅 `+ C` 部分 MISS
- `A + C` → 不完全匹配 `A + B` → **完全 MISS**
  - 但系统检测到公共前缀 `A`，将 `A` 持久化
  - 第三次 `A + D` → 完全匹配已持久化的 `A` → **HIT**

### 重要限制

- **"best-effort" basis** — 不保证 100% 命中
- **首次始终 MISS** — 缓存是请求过程中构建的
- **存活时间** — 缓存几小时到几天后自动清除
- **预热期** — 前 1-2 次请求为预热期，之后相同前缀即可命中

## 羽渡尘的缓存对齐策略

### 核心结论

**字典序对齐（v5.5 CacheAligner）是唯一真正有效的优化。**

### 为什么块对齐（v5.6 Chunk-aligned Truncation）被存档？

1. DeepSeek 缓存基于**前缀单元**匹配，不是逐块哈希——块对齐本身不提高命中率
2. 真实 prompt 系统提示词 ~1,500t + 工具 ~200t + 记忆 ~14t = ~1,715t
   - 碎片通常 < 768t（67.5%），不到对齐阈值
3. 即使碎片 > 768t，补 padding 到 1K 块边界只影响最后~300t
   - 这 300t 中还有用户查询的波动，缓存已无法保证
4. **边际收益 < 0.1%**

### 有效策略排序

| 优先级 | 策略 | 收益 | 实现 |
|:------:|:----|:----:|:----|
| 1 | system prompt 绝对稳定不变 | ~50% | 工程约束 |
| 2 | 记忆按实体 ID 字典序重排 | ~49% | `CacheAligner.align()` |
| 3 | 固定格式化模板（每行 `· entity: fact`） | ~1% | 模板化 |
| 4 | 块对齐补 padding | <0.1% | 已存档 |

### CacheAligner 接口

```python
aligner = CacheAligner(stable_prefix="【羽渡尘记忆】")

# 核心：字典序重排 → 固定模板 → 计算哈希
result = aligner.align(memory_results)

# 状态追踪
stats = aligner.stability_report()
# → {hits: 999, misses: 1, hit_rate: "99.9%"}
```
