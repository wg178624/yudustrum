# 控制论审计框架

将工程控制论三原则应用于软件/系统审计的通用方法。

## 三条审计维度

### ① 反馈回路（Feedback）

目标：验证系统存在有效的负反馈，且增益适中。

检查点：
- 系统出错时有没有机制捕捉并修正？
- 修正后信息是否固化（存羽毛/改配置）？
- 反馈太弱 → 系统发散（同一个坑反复踩）
- 反馈太强 → 系统震荡（过度敏感，不敢动作）
- 反馈是否收敛？即：纠偏一次后，同类问题不再发生

应用案例（羽渡尘审计 2026-05-26）：
- 用户纠偏 3 次（记事本/百科/画像分家），每次系统更新了
- 没有同一个坑踩两次的记录
- → 反馈增益适中，系统收敛 ✅

### ② 稳定性（Stability）

目标：系统在正常运行条件下不震荡、不漂移。

检查点：
- 核心指标是否稳定？（命中率、响应时间、错误率）
- 添加新功能后，老功能是否受影响？
- 系统有没有陷入"修了bug又引入新bug"的循环？
- 边界条件（满负载、数据增长）下是否仍稳定？

应用案例（羽渡尘审计 2026-05-26）：
- 缓存命中率 >99% → 稳定运行
- 595KB 实体索引加载需 12s → 有性能噪音，但不影响运行时稳定性
- → 整体稳定，有可量化的噪音 ⚠️

### ③ 鲁棒性（Robustness）

目标：意外情况下系统能兜底，不完全崩溃。

检查点：
- 每一个关键路径是否有多级 fallback？
- 非法输入/异常状态下系统行为可预期吗？
- 单点故障是否会导致全系统崩溃？

应用案例（羽渡尘审计 2026-05-26）：
- 四路召回：实体→图谱→FTS5→向量，每级兜底
- 审计脚本能检测 JSON 损坏
- Ghost feather 检测但不会自动修复 → ⚠️
- → 主要路径有兜底，边缘情况靠手动

## 跨领域适用

| 领域 | 反馈 | 稳定 | 鲁棒 |
|------|------|------|------|
| 电路 | 负反馈放大器 | 自激振荡 vs 稳定 | 电源纹波下正常工作 |
| 软件 | CI/CD / Code Review | 重构不改坏已有功能 | 非法输入不崩溃 |
| 管理 | KPI / OKR | 团队不内耗不崩 | 核心离职还能转 |
| 个人 | 别人说你错了你改 | 不被一个批评打倒 | 被辞还能找到工作 |

## 出处

工程控制论，钱学森，1954/1958。
