# v5 设计决策记录

## 架构演进：从 v4 → v5

### 为什么不用 Neo4j？

用户 VPS 只有 1 核 CPU，跑 Neo4j 太沉重。
选择 NetworkX（纯 Python 图库），内存运行，D 盘序列化 JSON。
135 节点 / 1414 边，加载 < 1 秒。

### 为什么反馈权重代替了完整的 RLHF？

Cognee 的 feedback_influence 机制的核心是一个计数器。每次命中 +1，未命中 -1。
完全够用。不需要 RLHF 的训练管线。

### 为什么冷热换页用 30 天阈值？

MemGPT 的 OS-like paging 提出 30 天作为默认冷热分界。
在我们 18 天羽渡尘历史的场景下，30 天内的实体都算活跃。
等羽渡尘运行 3 个月以上后，阈值可下调到 14 天。

### 分层字典序 v2（2026-05-23 第二版）

**问题：** 全局字典序虽然保证了缓存对齐，但把高频重要事实（RTX 4070 3分）和低频背景事实（Rust 0分）混在一起按字母排。高频事实被埋在列表中段。

**方案：** 按权重阈值（weight >= 1）切成两个兜：
- **核心高频兜**（活跃记忆）：权重 ≥ 1，内部字典序 → prompt 前面
- **次要背景兜**（背景记忆）：权重 < 1，内部字典序 → prompt 后面

**为什么可行：** 权重阈值 1 是稳定分界——实体至少被命中过一次才会进核心兜。一旦进兜就永远不掉，所以每次请求的核心兜内容稳定 → 前缀哈希稳定 → 缓存命中。

**收益：** 重要信息绝对靠前（防降智） + 每个兜内部哈希对齐（保缓存）。

**实现：** `CacheAligner.align(results, core_threshold=1)`

### 为什么语义密度蒸馏用正则匹配而非 LLM 分类？

每次查询前调 LLM 判断"这个查询属于什么密度"本身就是 token 开销。
用正则匹配：0 token 开销，毫秒级判定。
实测 5 类场景的匹配准确率 > 95%，无需 LLM。

### 为什么块对齐被存档？

参考 `references/cache-alignment-filter.md`：
1. DeepSeek 缓存基于前缀单元，非块哈希
2. 真实 prompt 碎片 < 768t，不值对齐
3. 边际收益 < 0.1%

### 为什么不用 PostgreSQL 替代 SQLite？

用户环境是 WSL（Debian），SQLite 零依赖。
且羽渡尘现存数据量 < 1MB，SQLite 足够。

## 目录结构决策

| 目录 | 用途 | 为什么不合并 |
|:----|:-----|:------------|
| feathers/ | 手写 Markdown | 人读的，用 Markdown |
| atoms/ | 机器提取的事实 | JSON，机器读写 |
| scenarios/ | 场景聚合 | 从 atoms 自动生成 |
| entities/ | 实体+权重+时间 | 独立因为每个更新频繁 |
| graph/ | 关系图谱 | NetworkX 序列化 |
| compact/ | 知识蒸馏 | LLM 生成的，定期更新 |
| symbols/ | 符号摘要 | 替代全文，省 Token |
| cold_storage/ | 冷存储 | 不活跃记忆，按需加载 |
| reports/ | 周报 | 人读的，Markdown/JSON |
| changelog/ | 变更历史 | 记忆版本控制 |
| snapshots/ | 快照 | 支持回滚 |
| auto-save/ | 自动存盘 | 每次会话结束自动存 |

## 版本号规则

- v1 → 核心存盘能力
- v2 → 搜索引擎（SQLite FTS5 + ChromaDB）
- v3 → 分层记忆（L0-L3）
- v4 → 符号化 + 实体 + 权重 + 时间窗口
|   |v5 → 关系图谱 + 压缩融合 + Agentic + 版本控制 + 密度蒸馏 + 缓存对齐 + 审计 + 去重 + 融合 + 字典序 + 固定格式
| v5.8 → 检索管线升级：去重(按entity)→字典序排序(第一优先级)→轻度融合(同实体合并)
|  | 输出格式绝对固定：【羽渡尘记忆】+·Entity:内容
|  | 抛弃核心/背景兜分层，改为全局字典序
