# TencentDB-Agent-Memory 架构参考

来源：https://github.com/TencentCloud/TencentDB-Agent-Memory (5.2k stars)
语言：TypeScript / Node.js
协议：MIT

## 核心架构（4级渐进管线）

### L0 — 基础存储层
- 原始对话/事件日志
- 不做处理，保留完整上下文
- 类似羽渡尘的 feathers/ 原始存档

### L1 — 记忆提取层（LLM驱动的场景理解）
- LLM 对话历史 → 结构化记忆
- 场景分割（Scene Segmentation）→ 把长对话拆成独立片段
- 实体/关系提取（命名实体识别）
- 同羽渡尘的 L1 提取器思路，但使用了更低成本的模型

### L2 — 记忆整合层
- 去重、合并、矛盾检测
- 时间线管理（关键创新）
- 跨场景的知识缝合

### L3 — 检索增强层
- 三层召回：关键字精确匹配 + 嵌入向量语义 + 混合查询
- 与羽渡尘的 RRF 混合排序等价但更系统化

## 关键设计点

1. **Scene Navigation（场景导航）**
   - 场景摘要优先注入上下文 → 需要细节时才加载全文
   - 类比羽渡尘的 brief 优先，但粒度更大（按场景/按片段）

2. **自动调度管线**
   - 不是每次对话都跑完整管线
   - 根据对话复杂度动态决定跑几层
   - L0（低价值/简单对话）→ 只存不处理
   - L3（高价值/复杂对话）→ 全量处理

3. **Checkpoint 快照**
   - 关键节点自动保存状态
   - 可回滚到任意 checkpoint
   - 羽渡尘已有对应的实现

4. **记忆工具指南**
   - 一组预定义的检索策略
   - 让 LLM 选择用什么策略查什么

## 与羽渡尘的异同

                 TencentDB           羽渡尘 v7.0
  语言           TypeScript           Python
  数据库         自定义/JSON          YantrikDB (Rust引擎)
  分层           L0-L3               13层(L0-H)
  检索引擎       关键字+向量+混合     向量语义
  矛盾检测       通过合并间接实现     显式支持(conflicts())
  场景导航       按场景分割           brief/full双存
  离线能力       内置                 内置

## 对羽渡尘的参考价值 (已落地或待落地)

  ✅ 已落地的:
  - L0-L3 分层的设计思路 → 羽渡尘的 13 层
  - 近因排名（Recency-aware ranking）→ RRF 混合排序
  - 场景导航 → brief 优先加载
  - 使用低频时通过快照而非重新计算的策略 → Checkpoint 快照

  ❌ 没拿的:
  - Scene Segmentation → 静态文件中不适用
  - LLM 驱动的 L1 提取 → 暂搁置（等羽毛 >200）
  - 自动调度管线 → 对于个人 VPS 没必要
