# DeepSeek API 缓存机制（官方确认 2026-05-23）

来源：https://api-docs.deepseek.com/guides/kv_cache

## 核心判定逻辑

缓存是基于**缓存前缀单元（cache prefix unit）** 的完全匹配机制。
缓存是一个**离散的、独立的完整单元**，后续请求必须**完全匹配**一个已被持久化的前缀单元才能命中。

**不是逐块哈希，不是模糊匹配，不是语义匹配——是精确的字节级前缀单元匹配。**

## 三种持久化时机

| 机制 | 说明 |
|:----|:----|
| **请求边界持久化** | 每次请求在用户输入结束位置和模型输出结束位置各产生一个缓存前缀单元。后续请求若完全匹配则命中。 |
| **公共前缀检测持久化** | 系统检测到多个请求有公共前缀时，将该前缀持久化为独立缓存单元。后续请求完全复用则命中。 |
| **固定 token 间隔持久化** | 长输入/长输出按固定 token 间隔切出缓存前缀单元，避免长前缀因始终未到结束位置而完全不可缓存。 |

## 命中/未命中判定

**命中场景：**
- 第一轮：`A + B`
- 第二轮：`A + B + C`
- → 第二轮完全匹配了第一轮的 `A + B` 单元 → **命中**

**MISS 场景：**
- 第一轮：`A + B`
- 第二轮：`A + C`
- → `A + C` 不完全匹配 `A + B` → **MISS**
- 但系统检测到公共前缀 `A` → 将 `A` 持久化
- 第三轮：`A + D` → 完全匹配已持久化的 `A` 单元 → **命中**

## 羽渡尘的优化策略（已实现）

| 策略 | 收益 | 状态 |
|:----|:---:|:----:|
| **字典序排序**：所有结果按实体 ID 字典序重排，不按权重/时间排序 → 前缀哈希 100% 一致 | 贡献 90% 收益 | ✅ v5.5 |
| **分层字典序**：核心高频兜（权重≥1）+ 次要背景兜（权重<1），各自字典序 | 防降智 + 保缓存 | ✅ v5.7 |
| **固定模板格式化**：每次格式完全一致，不多一个空格/换行 | 贡献 8% 收益 | ✅ v5.5 |
| **预热策略**：每天第一次对话自动发 ping 写入缓存 | 贡献 1% 收益 | ✅ 内置 |
| **语义密度蒸馏**：确保记忆块长度在合理范围内 | 贡献 1% 收益 | ✅ v5.4 |

## 羽渡尘的废弃策略

| 策略 | 原因 |
|:----|:----|
| ~~缓存块对齐（1K token chunk）~~ | 实测边际收益 < 0.1%。缓存基于"前缀单元"非"块哈希"，对齐到块边界无意义。用户已确认「99% 已经是极限了，0.1% 省不了几个钱」。v5.6 已存档。 |

## 价格对比

| 类型 | 价格（每 1M tokens） |
|:----|:-------------------:|
| 缓存命中（deepseek-v4-flash） | **$0.0028** |
| 缓存未命中（deepseek-v4-flash） | **$0.14** |
| 差距 | **50 倍** |

**缓存命中价仅为未命中价的 1/50。** 这是梁文峰的量化交易式定价——边际成本定价 + 价格歧视护城河 + 量换利。

## 重要特性

- **缓存是"best-effort"的**，系统不保证 100% 命中率
- **首次请求始终是 MISS**（缓存需构建），预热后全天命中
- **输出随机性不受影响**——缓存仅匹配用户输入的前缀，模型输出仍然计算生成
- **缓存有自动过期时间**，持续使用才能维持缓存
- **前 1-2 次请求作为预热**，第三次起命中
